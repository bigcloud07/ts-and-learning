- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
  - **4번**
    - **학습 로그 #1**에서 설계한 학습 전략의 2번 과정 진행시, 결과로 반환하는 person의 타입을 좁혀주지 않아 에러가 발생한다. 라고 문제를 정의했습니다.
    - 이후 AI에게 내가 문제를 정의한 표현이 정확하냐고 물었고 전체적인 의미는 맞지만 아래와 같이 더 정확하게 수정하면 좋겠다고 답변이 왔습니다.
      >`Person` 유니온 타입을 `Admin`, `User`로 안전하게 좁혀야 하는데, 타입 가드를 사용하지 않으면 person.role, person.occupation에 접근할 때 에러가 발생한다.
    - return 값을 안전하게 타입을 좁히는 부분을 설명할 때 타입 가드가 필요하다는 사실을 알았고 `is` 문법을 사용하면 되겠다고 판단이 되었습니다.
    - `isAdmin`, `isUser`에 타입 가드를 추가하고 나머지 과정을 진행했습니다. 
  - **5번**
    - 2번 과정 진행시, 사용자 필터 조건인 `criteriaKeys`의 속성을 age만 받아서 filterUser 함수를 구성하고 싶은데 현재는 User 전체의 속성만 받아오고 있어 에러가 발생한다. 라고 문제를 정의했습니다.
    - 이후 AI에게 내가 문제를 정의한 표현이 정확하냐고 물었고 아래와 같이 더 정확하게 수정하면 좋겠다고 답변이 왔습니다.
    > 사용자 필터 조건인 criteria는 age처럼 일부 속성만 받아야 하는데, 현재 User 전체 타입으로 선언되어 있어 { age: 23 } 같은 부분 조건을 전달할 때 에러가 발생한다.
    - 이 답변을 받고 age만 속성만 받을때를 염두해 둔 것이 아니라
    ```typescript
    filterUsers(
    persons,
    {
        age: 23
        occupation: 'Ball'
    }
    ).forEach(logPerson);
    ```
    같이 **일부 속성**을 사용자 필터 기준으로 삼아야 한다는 문제 요구 사항을 더 정확히 알 수 있었습니다.
    - `Partial`과 보너스 문제 요구사항인 `Omit`을 이용하여 문제를 해결 후 나머지 과정을 진행했습니다.

    - 결과적으로 원활한 문제 풀이를 하지 못하고 솔루션을 참고 했습니다.

      - 어떤 객체가 들어와도 더 편하게 key를 반환하는 `getObjectKeys()`를 구현한다.
    - 이후 첨부되어 있는 TS 공식문서를 봤더니 **함수 오버로드**에 대한 설명이 있었습니다.
    - 그럼 "`filterPersons` 함수를 오버로드 해서 고쳐야겠다!"라는 생각이 들었지만, 왜 고쳐야 하는거지? 라는 생각이 들고 이 부분부터 <u>**막힌거**</u> 같습니다.
    - 함수 오버로드가 단순한 문법처럼 보였고, 왜 사용해야 하는지 와닿지 않았습니다.
    - "왜 필요할까?"라는 생각을 가지고 솔루션을 확인했습니다.
    - 이후 솔루션을 기반으로 다시 문제 분석과 학습을 진행했습니다
      > - 문제의 핵심 요구사항은 personType에 따라 반환 타입이 `User[]` 또는 `Admin[]`로 달라지도록 만드는 것이었다.
      >- 솔루션을 통해 이 문제의 본질이 “입력값에 따라 반환 타입이 달라지는 함수”를 타입으로 표현하는 것이라는 점을 이해하게 되었다.
      >- 일반 함수로는 반환 타입이 `User[] | Admin[]`로만 추론되기 때문에, 호출 시점에서 정확한 타입을 알 수 없다는 한계가 있었다.
      >- 이를 해결하기 위해 함수 오버로드를 사용하면 personType 값에 따라 반환 타입을 명확하게 분기할 수 있다는 것을 알게 되었다.
      >- 추가적으로 Partial, Omit을 통해 criteria를 부분 속성으로 제한하고,
      >- Object.keys의 타입 문제를 해결하기 위해 제네릭과 keyof를 활용한 getObjectKeys가 필요하다는 것도 함께 학습했다.
